About

Moteye is a ubiquituos computing project at Ulm University developed by master students in 2014/15. In context of this project our concept of Gaze-Context-Mapping was developed. Here eye tracking technology is used in an automotive setting. The concept then was implemented as proof of concept in a prototype.



Concept



  • Motivation

    Have you ever been confused by the amount of controls in today's car cockpits? Noticed that taking your hand off the steering wheel to control a interface reduces safety in emergency situations? Have you ever rented a car and searched for a specific function?

  • Technology

    Looking through possibilities to stem this tide of output and input in a car, eye trackers show great potential. They are precise and have recently become small and affordable and thus allow easy integration in a automotive environment.

  • Idea

    How may eye trackers be used in an automotive setting? Can we unify driver input for the flood of controls for tertiary input devices such as entertainment systems? Increase safety by allowing full control from the steering wheel? Reducing the time required to find the correct control for a cockpit component?

  • Gaze Context Mapping

    Why don't you define physical components of a car cockpit as targets and map the user's eye gaze onto them? This provides the possibility to select these with a short glance. To interact with them a single control joystick is placed on the steering wheel allowing adjistments while keeping your hands on the wheel.

Technical



Moteye uses a distributed system architecture of nine applications. These may run on independant computer systems. Connection is establishd via internet protocols for the purpose of this prototype.

On top the modules are shown which create input to the system: namely the eye trackers as well as the joystick control. Any number of eye trackers can be attached to the system. However due to driver limitations only seven eye trackers can be used per driver instance. By using multiple instances of Trackteye this limitation can be mitigated.

Gaze data is received and preprocessed by the Trackteye module written in C# and then handed over to the Java part of this module. There it is processed and sent to the Coreteye module. Joystick input comes in via the serial port of the Arduino, is parsed, and then also directed to Coreteye.

Coreteye merges the gaze and button data and then runs selection logic on it. This logic detects which component, if any, has been selected and decides where to dispatch the event to. Coreteye also handles the user profile functions of the system, allowing multiple profiles to be managed and used according to the user's preferences.

The components, including the radio called Voxteye, the mirror called Mirteye, and the hands free call system called Callteye react to incoming events. As can be seen by the implementation of the Android Callteye app, components are not restricted to pure Java applications. Each component is totally free to implement any desired functionality build on top of a small framework that handles the reception of events from Coreteye.
